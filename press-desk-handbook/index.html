04 · Published vs Draft Tabs

The two coloured pills at the top of every document.

This confuses everyone. Here's what's happening.

Inside any document editor
● Published ○ Draft ───────────────────────── ▲ ▲ │ │ │ └─► The DRAFT version. Your edits live here while │ you're working. Auto-saved every few seconds. │ Not visible to the public site until Published. │ └─► The PUBLISHED version. The version that's live on recurr.co right now. (Once it has been published at least once.) STATES OF THESE TWO PILLS: ● Published ○ Draft No changes since last publish. (Both pills filled = same content.) ● Published ● Draft You've made edits. Draft is ahead of published. Hit "Publish" to push them live. (no pill) ● Draft Brand-new doc. Never published. Hit "Publish" to take it live for the first time. ● Published (no pill) Published doc with no pending edits. CLICKING A PILL: ───────────────── Click "Published" → you see the live version (read-only). Click "Draft" → you see (and edit) the working version.
Plain English

Published is what readers see. Draft is what you're working on. Sanity keeps both versions side by side. The Publish button moves your draft into the published slot.

05 · The Mode Dropdown (Top-Right Corner)

Published / Drafts / Scheduled / Releases / New release.

This dropdown changes which version of the dataset the entire Studio is showing.

The dropdown, expanded
┌─────────────────────────────┐ │ ● Drafts ▼ │ ◄──── current mode shown in pill ├─────────────────────────────┤ │ ● Published │ ◄──── view the live state of everything │ ● Drafts (current) │ ◄──── view drafts (default working mode) │ ───────────────── │ │ 📅 View Scheduled Drafts │ ◄──── enterprise feature (skip) │ 📅 View Content Releases │ ◄──── enterprise feature (skip) │ + New release │ ◄──── enterprise feature (skip) └─────────────────────────────┘

For day-to-day editorial work, leave it on Drafts. That's the default. You're editing drafts. When you publish a doc, your draft gets pushed to the published version.

Switch to Published only if you want to see what the site is currently showing — useful for diff-checking before a publish.

The three lower entries — Scheduled Drafts, Content Releases, New release — are part of Sanity's enterprise Releases feature. We don't use them. Ignore them.

06 · The Editorial Workflow (Kanban)

Brief → Draft → In Review → Approved.

The Kanban is a separate tool, accessed via the Workflow tab in the top nav. It's a board with four columns. Each card on the board is a Press Article (or a Newsletter).

The Kanban board
┌──────────────┬──────────────┬──────────────┬──────────────┐ │ BRIEF │ DRAFT │ IN REVIEW │ APPROVED │ ├──────────────┼──────────────┼──────────────┼──────────────┤ │ • Idea card │ • Body in │ • Reviewer │ • Signed │ │ • Title set │ progress │ reading │ off by │ │ • Slug set │ • TLDR set │ • Comments │ EIC │ │ • Excerpt │ • Cuts │ being │ • Ready │ │ • Pillar │ drafted │ left │ to ship │ │ • Body=∅ │ │ │ │ │ │ │ │ │ │ drag→ │ drag→ │ drag→ │ hit │ │ │ │ │ PUBLISH ─► │ └──────────────┴──────────────┴──────────────┴──────────────┘ │ └─► leaves the Kanban, appears at /press

Who can move what

  • Anyone can drag Brief → Draft → In Review (and back).
  • Only the Editor-in-Chief can drag In Review → Approved.
  • The Publish button (separate) is what actually puts the piece on the public site.
Approved ≠ Published

Approved means "the EIC says ship it." Published means "it's actually live." The Publish action is a separate click. Until somebody hits Publish, the piece is sitting in Approved, ready, but not yet on recurr.co.

07 · Your Mental Model — Reviewed

Where your assumption is right, and where it isn't.

Your version

"Brief → when content has only a brief. Draft → when somebody is drafting. In Review → when someone is reviewing. When I move it to Approved, it should automatically get published."

The truth

  • Brief / Draft / In Review — your description is correct.
  • Approved — your description is correct except for the auto-publish.
  • Approved does NOT auto-publish. You still have to click the Publish button on the document itself.

Why? Because the Kanban and the Publish action are independent in Sanity. The Kanban is editorial workflow. Publishing is the Sanity primitive that makes the doc visible to the public site. They were designed separately.

What "Approved" actually means
APPROVED ────────────────────────────────────────────────────► PUBLISHED │ ▲ │ (manual action) │ │ │ └─► Open the doc. ──► Hit "Publish" button (top right). ─────┘ One click.

Could we make Approved auto-publish?

Yes — by adding a Sanity Studio action that auto-publishes when state moves to Approved. Not built today. Open thread.

08 · What Happens After Publish

Where a piece lives once it's live.

When you hit Publish on an Approved piece, two things happen: the doc becomes visible on the public site, and its workflow card leaves the Kanban. Published pieces don't disappear — they just live somewhere else.

After publish
PUBLISH BUTTON CLICKED │ ┌─────────────┼─────────────┐ ▼ ▼ ▼ ┌──────────────┐ ┌────────────┐ ┌────────────────┐ │ PUBLIC SITE │ │ ARTICLES │ │ KANBAN │ │ │ │ LIST │ │ │ │ recurr.co │ │ │ │ Card removed │ │ /press │ │ Press Desk │ │ from Approved │ │ /press/│ │ → Articles │ │ column. │ │ │ │ → All │ │ │ │ Sorted by │ │ articles │ │ (Workflow ends│ │ publishedAt │ │ │ │ at Approved.) │ └──────────────┘ └────────────┘ └────────────────┘

To see published pieces in the Studio

  • Press Desk → Articles → All articles — sorted by publish date, newest first.
  • Press Desk → Articles → By theme — pick a theme, see its pieces.
  • Press Desk → Releases — pick a weekly Release to see what shipped that week.

To unpublish a piece

Open it. Hit the three-dot menu top right. Choose Unpublish. The piece comes off the public site. The doc remains as a draft — you can re-publish later.

12 · Themes & Releases

The umbrella, and the bundle.

How they relate
┌─────────────────────────────────────────────┐ │ MONTHLY THEME e.g. "AI Will Eat SaaS" │ │ ───────────── │ │ Sub-theme W1 Sub-theme W2 │ │ Sub-theme W3 Sub-theme W4 │ └─┬──────────────────────────────────────────┬─┘ │ │ ▼ ▼ ┌─────────────────┐ ┌─────────────────┐ │ WEEKLY RELEASE │ │ WEEKLY RELEASE │ │ May W1 │ │ May W4 │ │ │ │ │ │ Bundles: │ │ Bundles: │ │ • Essay │ ... ... │ • Essay │ │ • Co-authored │ │ • Co-authored │ │ • Research │ │ • Research │ │ • Newsletter │ │ • Newsletter │ └─────────────────┘ └─────────────────┘ │ │ ▼ ▼ 4 articles 4 articles ship together ship together
  • Theme — one per month. The umbrella. Lives in Press Desk → Library → Themes.
  • Release — one per week of that month. Bundles articles to ship as a coordinated drop.
  • An article references a theme via the Theme field. It can also reference a Release.
  • Both theme and release are optional. Standalone pieces ignore both.

Pratik's question: "Weekly Release is basically a sprint, right?" Yes. That's the right mental model. A Release is a weekly editorial sprint — a labelled bundle of pieces that ship in the same week.

Don't confuse this with Sanity's "Releases" tab

Our Weekly Release is a custom document type we built. Sanity also has its own enterprise feature called Releases (the top-nav tab). They are unrelated. See section 18.

13 · The Newsletter

How the weekly digest works.

The newsletter is a separate document type from articles. Don't pick "essay" format on a Press Article and call it a newsletter.

Newsletter doc structure
┌────────────────────────────────────────────────┐ │ NEWSLETTER (one per week) │ ├────────────────────────────────────────────────┤ │ weekLabel "May W1" │ │ subject "May W1 — Interface…" │ │ preheader Hidden inbox preview line │ │ intro Personal note (150 words max) │ │ featuredArticles → references Press Articles │ │ signals → references Signal items │ │ outboundBody What Substack actually gets │ │ substackUrl (auto-set after publish) │ └────────────────────────────────────────────────┘

Where to write it

Press Desk → Newsletter. Each entry is one week's edition.

Where it goes

Substack is the public channel. The plan is for an automated worker to pick up Approved newsletter docs and post them to Substack — the wiring is built but not yet deployed. Until then, copy-paste from Sanity to Substack.

What goes in it

Mental models for venture businesses. Not tactical content. Each week's newsletter pulls in the Newsletter cut companion field from that week's articles, plus a fresh editorial framing.

14 · Signals

The automated content firehose.

A Signal is a small, atomic piece of curated content — a funding announcement, a positioning move, a launch, an industry datapoint. The Signal Engine collects them automatically; we approve, we post.

Signal lifecycle
SOURCES (RSS, Twitter lists, HN, funding APIs, etc.) │ │ daily ingest, automated ▼ ┌─────────────┐ │ QUEUED │ ──► Editor reviews ──► ┌──────────┐ │ │ (kill or approve) │ APPROVED │ │ scored by │ │ │ │ relevance │ │ scheduled│ │ × novelty │ ◄── back if revise │ to post │ └─────────────┘ └─────┬────┘ │ │ posting time ▼ ┌──────────┐ │ POSTED │ ──► @recurrco on X │ │ │ visible │ ──► /press/signals │ on site │ (living archive) └──────────┘ (Or, anytime: "Killed" = rejected. Hidden from queue, not deleted.)

Where to find them

  • Press Desk → Signals → Queue (ready to review) — daily review pass lives here.
  • Press Desk → Signals → Posted — history of what's gone live.
  • Press Desk → Signals → All signals — everything, including killed.

The two outputs

  1. Twitter / X — @recurrco. Posted signals go out as tweets, automatically.
  2. The signals page — recurr.co/press/signals. A living archive that compounds.
15 · What is "Workflow Metadata"?

You shouldn't normally see this folder. Here's what it is.

Workflow Metadata is a behind-the-scenes data structure used by the Kanban plugin. Every article on the Kanban has a hidden metadata sibling that holds its current state (Brief / Draft / In Review / Approved). That's it.

The article and its hidden metadata sibling
┌──────────────────────────┐ ┌──────────────────────────────┐ │ PRESS ARTICLE │ ◄────── │ workflow.metadata │ │ │ refs │ │ │ _id: post.foo │ │ _id: workflow-metadata. │ │ title: "..." │ │ post.foo │ │ body: [...] │ │ │ │ ... │ │ state: "draft" │ │ │ │ assignees: [user-1] │ │ │ │ orderRank: "0|hzzzzz:" │ └──────────────────────────┘ └──────────────────────────────┘ What you edit What the Kanban reads (the actual article) (state, position on board)

You almost never need to open these directly. The Kanban manages them. They're in the Content nav by accident — we'll hide them in the next config push.

If you ever do click into one: don't edit it. Close it. The Kanban will resync next time you load it.

16 · What is "AI Context"?

A briefing document for Sanity AI Assist.

AI Context is a place where you write a briefing for Sanity's AI Assist features. When you ask AI Assist to draft a Twitter thread or rewrite an excerpt, it reads the AI Context first to understand the brand voice.

How AI Assist uses the context
YOU AI ASSIST ─── ───────── Click "AI Assist" on a field │ ▼ "Draft a Twitter thread from this body" │ ▼ ┌──────────────────┐ │ Read AI Context │ ◄─── voice │ Read this body │ guidelines │ Apply both │ brand tone └────────┬─────────┘ examples ▼ Generate output │ │ ▼ │ Draft appears in field ◄───────────────────┘

What to put in AI Context

  • Voice guidelines: "opinionated, not neutral. Framework-first. Plain language about hard ideas."
  • Format examples: "Twitter threads: hook first, last tweet ends with a pointed question."
  • Anti-patterns: "no hashtags. No 'thread 🧵'. No 'In this thread'."
  • Brand-specific terms: how we name pillars, formats, themes.

One AI Context document per project is enough. We'll seed a v1 with the editorial framework. You can edit it freely — it improves AI Assist outputs immediately.

You only need to look at AI Context if (a) AI Assist is generating off-brand drafts, or (b) you want to teach it a new pattern.

17 · Sanity Tasks (Ignore for now)

Built-in Sanity feature. We don't use it.

Sanity has a built-in task-tracker on every document. We don't use it. Use the Kanban for editorial work, and Plane (Outwork) for project management. Don't create Tasks.

Where Sanity's Tasks lives
ANY DOCUMENT ──────────── Top-right of editor: ┌──────────┬──────────┬──────────┐ │ Comments │ Tasks │ ... │ └──────────┴──────────┴──────────┘ │ └─► Per-document task tracker. Independent of our Kanban. Has its own statuses (To Do / Done). No relationship to Brief / Draft / In Review / Approved. USE INSTEAD: • Kanban tab in top nav (for editorial state of the article) • Plane / Outwork (for "do X by Y" project tasks) • Comments on the doc (for inline feedback)

Why we don't use Sanity Tasks: it duplicates the Kanban without integrating with it. Two tracking systems is one too many.

19 · Order Buttons in the Workflow Tab

"Set unique order for two documents" / "Reset order for all 17 documents."

Bottom-left of the Workflow Kanban. Almost never needed.

What they do
THE PROBLEM: Every card on the Kanban has a position (so cards stay in the order you arranged them). When two cards somehow end up with the SAME position, dragging them gets weird. THE FIX: These two buttons reassign positions: ┌─────────────────────────────────────────────┐ │ Set unique order for 2 documents │ │ │ │ Find the duplicates and give each a fresh │ │ unique position. Surgical fix. │ └─────────────────────────────────────────────┘ ┌─────────────────────────────────────────────┐ │ Reset order for all 17 documents │ │ │ │ Wipe positions and reassign all of them │ │ fresh. Heavy reset. Use only if "Set │ │ unique" doesn't unstick the board. │ └─────────────────────────────────────────────┘

If the Kanban is behaving normally, ignore these. They appear because the plugin detected the situation. Pressing them won't lose any data — they only re-stamp ordering.

20 · Comments & Review

How feedback flows.

Every document has a Comments panel in the right sidebar. Click into a field, leave a comment, the author sees it.

  • Reviewer opens the doc, reads.
  • Comments can be on a field ("this excerpt undersells the piece") or on a selection in the body (highlight text, comment).
  • Author resolves comments by editing or replying.
  • Once resolved, the reviewer drags forward (Approved) or back (Draft) with a summary comment.
One rule

Don't comment and leave. Always move the piece to the next state when you're done reviewing.

21 · Publish Checklist

Before hitting Publish.

  • Title — final, not a working title
  • Slug — final (it won't change after publish)
  • Excerpt — under 280 characters
  • TLDR — under 160 characters, one sentence
  • Cover image — present, or accept the listing-card hit
  • Body — proofread end to end
  • Pillar — set
  • Byline or Authors — set
  • Format — set
  • Theme — set if there's an active monthly arc, blank if standalone
  • Distribution: Twitter thread, LinkedIn cut, newsletter cut written
  • SEO title — only if it differs from the editorial title
  • Published At — set to the scheduled time

When everything's green, hit Publish in the upper-right of the document. The site re-validates within a couple of seconds.

23 · When You're Stuck

Quick answers.

My Approved piece isn't on the site.
Approved doesn't auto-publish. Open the doc, hit Publish (top right).
What's the difference between Published and Draft?
Published is what readers see. Draft is what you're working on. Hit Publish to push your draft live. See section 4.
What's the Releases tab in the top nav?
Sanity's enterprise feature. We don't have it. Ignore the tab. See section 18.
What's "AI context"?
A briefing for AI Assist. Tells it our brand voice. See section 16.
What's "Workflow metadata"?
A behind-the-scenes structure used by the Kanban. Don't edit. See section 15.
What's the difference between a Theme and a Release?
Theme = monthly umbrella. Release = weekly bundle of pieces shipping together. See section 12.
My piece won't publish.
Check Pillar, Byline, Format are all set — those are required. Then check you're in Approved state.
I broke something.
Sanity has Document History. Revert from there.
