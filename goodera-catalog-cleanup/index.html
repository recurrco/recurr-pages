Cleaning the catalog before we build the smart layer

A plain-English walkthrough of the data cleanup we need to do first — so that filters, search, recommendations, and the Catalog Room actually work.

Scope: Activity catalog (1,094 activities · 2,296 variants)  ·  Based on: the VV domain schema + the June activity export  ·  For: Vishesh, Shubham, Karan

The catalog isn't broken — it's inconsistent. The same fact gets written several different ways, real categories hide behind the word "others," and some fields are half-empty. None of this stops the catalog working today — but all of it blocks the smart features (recommendations, personalization, the Catalog Room). You can't build intelligence on messy inputs.

The health check

Half of it is already clean. The mess is concentrated.

✓ Already solid
  • Business Unit only 2 clean values
  • Event Format 4 clean values
  • Duration all numeric, sensible range
  • Min Participants all numeric
  • Turnaround (TAT) all numeric
✗ Needs work
  • Volunteer Output 48% blank + free text ("Best", "2–3 cards")
  • Focus Areas real categories hidden as "others" (8% of rows)
  • Material Cost 35% blank + a few $100,000 typos
  • Activity Names 46 duplicate pairs — same item, different spelling

The path

Build the foundation, bottom-up

These are done in order. Each step stands on the one below it — skipping ahead means rebuilding.

↑   build upward   ↑
6
Lock it can't rot again
Rules at data-entry: pick a category from a list, no free-typing. Enforced only once the data already obeys.
5
Merge duplicates of the same fact
One fact, one home. Retire the messy free-text "volunteer output" and keep only the structured version.
4
Fill the gaps that matter
The 35% missing material cost, the blank outputs — fill with values from the agreed clean set.
3
Normalize every value
Map each messy value to its one canonical form: kit / Kit / kits → "kit". Fix the $100k typos.
2
Dedupe the catalog
Merge the 46 duplicate activities (EduCards = Educards) into single items.
1
Agree the official lists — FIRST the real decision
The canonical focus areas, output units, and tags — and break the "others" pile open into real categories. Everything above depends on this.
0
Snapshot & agree "what good looks like"
Freeze a copy of today's data before touching anything.

Why this exact order

Each layer needs the one under it

Can't dedupe before the vocabulary — deciding two items are "the same" needs agreed rules.
Can't normalize before dedupe — otherwise you clean the same duplicate twice.
Can't fill gaps before normalizing — you'd fill blanks with values that don't match the clean set.
Can't merge double-storage before filling — you might delete the only copy that held the data.
Lock it last — you only enforce rules once the data already passes them.

Step 1 is a business decision, not a technical one

The "others" trap

111

activities are tagged only "others"

That's 111 items the catalog literally cannot categorize, filter, or recommend — invisible to any smart feature.

"Others" isn't a category — it's a pile of missing categories. Cleaning it means opening the pile and naming what's really inside. That's a call the product team owns, and it's exactly the "rich tagging" the Catalog Room is built on.

Decide the canonical focus areas, output units, and tag vocabulary once — and ~80% of the mess collapses on its own, because most of it is just the same handful of real values written carelessly.

Vocabulary → Dedupe → Normalize → Fill → Merge → Lock

Foundations first, features second. Everything smart — recommendations, personalization, the Catalog Room — is a feature that only works once this foundation is solid.
