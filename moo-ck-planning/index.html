Materials Control

One tool for everything materials — buy · receive · pay · make · stock · dispatch — fed by Rista, decided by standard inventory rules. v2 · 21 Jul 2026

The one rule. This tool never writes to Rista — and it can't: our Rista key is read-only. It reads Rista + the warehouse, works out what should happen, and suggests it. A human keys the PO / transfer into Rista; the tool reads it back to confirm. read → suggest → remind → verify.

This isn't a new problem — it's textbook par-level (min/max) stock control + MRP + procure-to-pay. We use the standard names, not invented ones. Everything is built around one idea: the software works out the number; the person confirms with one tap.

The lifecycle

SOURCE ─▶ BUY ─▶ RECEIVE ─▶ PAY ─▶ TRANSFORM ─▶ STOCK ─▶ DISPATCH ─▶ VERIFY vendors PO GRN + bills + repack + par & indent → read & terms (sug- invoice bank production runway transfer → Rista gested) capture recon (we log it) board receive back

Vendors → Central Kitchen (CEN) + Central Store (CS) → 5 stores (HSR · IDR · JP · WFL · DOD).

The tool = 5 areas

Make

repack (25kg→2kg) + recipes (juices, oat base)
Send

Pay

vendor credit limit, used, pending, available

The three jobs underneath

Buy — Procure-to-Pay (Vendor → Central)

Show only materials below their Min, order quantity already worked out, vendor already attached. A material can have several vendors — we show each one's price, MOQ, pack size and credit headroom so the person picks. Receiving = snap the invoice, tick the pre-filled checklist, edit only the shortfall. Bills carry an app-owned due date + paid/unpaid, reconciled from the bank feed. Over time the GRN cost history tells us the true price of everything.

Order qty = max(0, Par − (on-hand + on-order)) ← never re-orders the same gap Credit headroom = credit limit − (billed − paid)

Make — Transformation (inside Central)

Two kinds, both things Rista can't record — so we log them: repack / bulk-break (1× 25kg oats → 12× 2kg packs, with a yield %) and recipe production (raw → juices / soaked-oat base / granola, shelf-life bound). What we make feeds Send; what it consumes feeds Buy.

Send — Dispatch (Central → Stores)

A two-sided handshake: the store asks (indent) → the kitchen accepts then dispatches → the store receives. The plan splits into Prepared (from Make) and Direct (shipped as-is from the store).

dispatch = max(0, Par − store stock) + ceil(velocity_7d × buffer_days) + indent capped by what the kitchen / store can supply

How we built it (decisions)

  • One responsive web tool to start — works on phone / tablet / desktop. (A dedicated floor phone app can split out later.)
  • English first (Hindi/Kannada + WhatsApp alerts are the stand-out next step — no competitor does it).
  • Credit = visibility only — show the limits and what's pending; no auto-picking of vendors yet.
  • Read-only Rista + we own our own writes — enforced by the key, safe by design.

Known gaps (be honest)

  1. Recipe/BOM quantities were manual — but GET /inventory/items may now expose them; one probe could remove the spreadsheet.
  2. Vendor lead time isn't stored anywhere — seed manually, derive from order→receipt history later.
  3. Bank reconciliation works, but vendor payments need tagging before credit headroom is exact.
  4. Repack & production aren't Rista events — we log them and post a stock adjustment by hand.
