Posting MOO Direct Orders into Rista — Why & How

We are launching our own ordering channel (our app, moo.kitchen) and we want each order to land in Rista exactly the way a Swiggy, Zomato, or walk-in order does today — same invoice, same reports, same settlement view — just tagged as our own channel. Rista stays the single system of record for all sales; we simply add one more channel into it via the API.

How the order reaches us (“how are you getting this order?”)

  1. Customer places and pre-pays the order on moo.kitchen (online payment via Razorpay).
  2. Our backend re-derives and validates the price server-side, then saves the order in our own database as the source record, with a stable order id.

How we post it into Rista (“how does it get onto Rista?”)

On payment success, our service calls POST /sale with:

  • branchCode — the fulfilling outlet (HSR / IND / BNG / CEN)
  • channel — a dedicated price channel for us (e.g. “MOO Direct”), so the order is priced and reported as a distinct channel, like Swiggy / Zomato / Walk-in
  • items[] — each line carries skuCode, quantity, unitPrice, itemAmount, tax, itemNature, and any add-on options[]
  • payments[] — the Razorpay collection recorded against the sale (mode, amount, reference)
  • sourceInfo — the block built for external orders:
    • invoiceNumber = our order id → drives idempotency (a retry returns 409 “already processed”, never a duplicate sale)
    • source = "moo.kitchen", isEcomOrder: true, isEditable: false (lock the order once created)
    • callbackURL = our endpoint, so Rista can push status (Accepted / Rejected / Prepared / Dispatched / Delivered) back to us
The gate — we never post a bad order

This is the important part for data integrity. Before an order JSON is ever built, every SKU in it is already proven to exist and be active in Rista. Our catalog system continuously reconciles our menu against the live Rista catalog and assigns each SKU an orderability status with a 1:1 link to the Rista item code. The app only ever shows — and only lets a customer buy — SKUs that are currently mapped and live in Rista. So a POST /sale from us will only ever contain SKU codes Rista recognises. No phantom items, no unmapped codes.

Why this is low-risk for you

  • Idempotent — the same invoiceNumber can never create two sales.
  • Locked — isEditable: false, so the record can't be silently mutated.
  • Reversible — if we ever post in error, we void via POST /sale/status (Voided).
  • Non-aggregator only — this is our own direct channel; it does not touch the Swiggy / Zomato sync at all.

The outcome we're after

A MOO Direct order shows up in your invoice list / reports / settlement next to Swiggy, Zomato, and Walk-in — and our warehouse, which already syncs Rista sales every 15 minutes, picks it up the same way it picks up everything else. One system of record, one more channel.
